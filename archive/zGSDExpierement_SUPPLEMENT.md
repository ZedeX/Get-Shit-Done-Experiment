## 验证环节设计

### 验证环节目标

验证环节的核心目标是：**确保产出物不仅仅被"编写出来"，而是真正能工作、能达到验收标准**。

### 验证环节时间占比

- 执行阶段: 60%
- 验证阶段: 40% (**必须保证充足时间**)

---

## 验证环节内容 (V.1 - V.5)

### V.1 真实数据/测试用例准备

**目的**: 准备真实的、有代表性的测试输入

**要求**:
- [ ] 代码开发任务: 创建真实的测试输入文件/数据
- [ ] 数据分析任务: 使用真实的(或真实模拟的)数据集
- [ ] 创意生成任务: 准备真实的需求背景和评估标准

**T1示例**:
- 准备多种尺寸PNG (小/中/大/超大)
- 准备多种格式PNG (RGB/RGBA/灰度)
- 准备嵌套目录结构
- 至少10-20个测试文件

---

### V.2 真实运行/执行测试

**目的**: 真实运行产出物，验证其功能

**要求**:
- [ ] 代码开发任务: 真实运行脚本/程序
- [ ] 数据分析任务: 真实运行分析脚本
- [ ] 创意生成任务: 真实评审产出物质量

**必须保存的证据**:
- 完整的运行LOG (stdout/stderr)
- 真实生成的输出文件
- 错误信息(如有)
- 性能数据(耗时/资源使用)

---

### V.3 功能验证检查清单

**目的**: 逐项验证验收标准

**要求**:
每项验收标准必须有:
1. **验证方法** - 如何测试这项标准
2. **验证结果** - PASS/FAIL/部分通过
3. **验证证据** - 截图、LOG片段、文件样本

**T1示例检查清单**:

| 验收标准 | 验证方法 | 验证结果 | 验证证据 |
|----------|----------|----------|----------|
| 1. 脚本可正常运行 | 运行`--help` | ✓ PASS | `gsd_help.log` |
| 2. PNG正确转换为WebP | 检查输出文件 | ✓ PASS | `output_gsd/` 目录 |
| 3. 文件名保持一致 | 对比输入输出文件名 | ✓ PASS | 文件列表 |
| 4. 转换质量可配置 | 测试quality=85/90/95 | ✓ PASS | 生成的文件大小对比 |
| 5. 支持RGBA透明图片 | 检查alpha_*.webp | ✓ PASS | 文件存在 |
| 6. 递归处理子目录 | 检查subdir/ | ✓ PASS | 文件存在 |

---

### V.4 质量量化评分

**目的**: 对各项质量指标进行量化评分，**每项评分必须有具体证据**

**评分原则**:
- 1-3分: 不满足要求，有具体证据
- 4-6分: 基本满足，有改进空间，有具体证据
- 7-9分: 良好满足，有具体证据
- 10分: 完美满足，有具体证据

**评分证据链要求**:

每项评分必须包含:
1. **观察到的事实** - 具体看到了什么
2. **与标准的对比** - 如何符合或不符合
3. **具体证据位置** - 文件路径/LOG行号/截图

**T1评分证据链示例**:

#### 评分项: 用户体验 (GSD模式)

**观察到的事实**:
- 运行时显示进度: `[1/19] 转换: filename.png`
- 完成后显示统计: `转换完成! 成功: 19/19`
- --help有完整描述和2个使用示例

**与标准的对比**:
- 进度显示: ✓ 有 (标准: 期望有)
- 完成统计: ✓ 有 (标准: 期望有)
- 帮助信息: ✓ 详细 (标准: 期望清晰)

**具体证据位置**:
- `gsd_run.log` - 第1-22行
- `gsd_help.log` - 第1-20行

**评分**: 10/10

### V.5 验证报告生成

**目的**: 生成结构化的验证报告，包含所有证据

**验证报告必须包含**:

```
## 验证概览
- 实验编号
- 验证时间
- 验证人员

## 真实运行证据
- 运行命令
- 完整输出LOG
- 生成的文件列表

## 功能验证结果
- 验收标准检查清单 (带证据)
- 每项的PASS/FAIL状态

## 质量量化评分
- 各项评分
- 每项评分的证据链
- 总分

## 问题与改进建议
- 发现的问题
- 改进建议
```

## 各任务类型的验证重点

### 代码开发任务 (T1-T3)

**验证重点**:
1. ✓ 真实运行程序
2. ✓ 功能测试 (正常路径+边界路径)
3. ✓ 错误处理测试 (错误输入/异常情况)
4. ✓ 性能测试 (时间/内存/资源使用)
5. ✓ 代码质量审查 (可读性/可维护性)

**必须测试的输入**:
- 正常输入
- 边界情况 (空输入/超大输入)
- 错误输入 (不存在的文件/无效参数)
- 特殊情况 (RGBA/特殊字符/嵌套目录)

---

### 数据分析任务 (T4-T6)

**验证重点**:
1. ✓ 真实加载和分析数据
2. ✓ 计算结果准确性验证 (手动验算样本)
3. ✓ 分析完整性检查 (是否覆盖所有要求)
4. ✓ 可视化输出检查 (图表是否正确)
5. ✓ 结论合理性验证 (是否有数据支撑)

**必须验证的数据点**:
- 手动验算3-5个关键计算
- 检查异常值处理
- 验证数据聚合逻辑
- 确认趋势分析合理性

---

### 创意生成任务 (T7-T9)

**验证重点**:
1. ✓ 完整度检查 (是否覆盖所有要求)
2. ✓ 质量评估 (多人评审/客观标准)
3. ✓ 风格一致性检查
4. ✓ 创新性评估
5. ✓ 实用性验证 (是否真的可用)

**创意任务验证方法**:
- 建立客观评分标准
- 至少2人独立评审
- 逐项对照需求清单
- 提供具体改进建议

## 验证不通过的处理流程

### 验证失败的判定

满足以下任一情况即判定为验证失败:
1. 核心功能无法工作
2. 验收标准通过率 < 70%
3. 质量评分 < 5/10 (平均)
4. 缺少必要的验证证据

### 处理流程

```
验证失败
    ↓
记录失败原因和证据
    ↓
决定是否重试
    ↓
┌─────────────┬─────────────┐
│   重试      │   继续      │
│ (有希望)   │ (时间限制)  │
└──────┬──────┴──────┬──────┘
       ↓             ↓
  修正后重试      标记为失败
  记录重试过程    记录失败原因
```
